# Security Policy

This policy applies to the k8s-diag-agent repository and its workflows for live evidence collection, artifact generation, LLM interactions, and adaptation proposals. It defines safe practices around secrets, private context handling, and automation boundaries.

## Scope and Goals

- Prevent secrets or private cluster identifiers from leaking into source control or LLM prompts.
- Protect credential confidentiality for providers (OpenAI, llama.cpp).
- Bound live evidence collection and redaction to user-approved contexts.
- Disallow autonomous cluster mutations without explicit operator approval.
- Define hygiene for runtime config, snapshots, runs, and logs.
- Require security review for dependency or provider changes.

## Secrets and Credential Handling

- All provider credentials (e.g. `LLAMA_CPP_API_KEY`, `OPENAI_API_KEY`) must be set via environment variables.
- Do not commit secrets or credentials to Git. `.gitignore` excludes `*.local.json`, snapshots, runs, and logs.
- Use private local config (`runs/*.local.json`, `snapshots/*.local.json`) for cluster-specific settings; these files must remain untracked.

## Live Evidence Collection Boundaries

- Capture live snapshots only for contexts defined in untracked `*.local.json` configs.
- The private-context checker (`ban-private-contexts` pre-commit hook) blocks any committed private cluster names.
- Do not collect or retain credentials, kubeconfig tokens, or raw private metadata beyond typed `ClusterSnapshot` fields.

## Redaction Rules

- Logs, artifacts, diffs, digests, and prompts must not contain secrets or private context values.
- Follow redaction standards in [docs/logging-policy.md](docs/logging-policy.md).
- Strip or anonymize private identifiers (cluster names, namespaces, node names) before sending data to LLM providers.

## Provider Boundary Rules

- Route all LLM interactions through the provider seam (`assess-snapshots`, `assess-drilldown`).
- **llama.cpp**:
  - `LLAMA_CPP_BASE_URL`, `LLAMA_CPP_MODEL`, and optional `LLAMA_CPP_API_KEY` control access.
  - The service endpoint must be under operator control; do not expose to public networks.
- **OpenAI-compatible providers**:
  - `OPENAI_API_KEY` and other `OPENAI_*` env vars guard access.
  - Never send private cluster details or credentials to external services.
- Default to local or offline analysis when provider configuration is missing.

## No-Autonomous-Cluster-Mutation Rule

- The agent must never perform direct mutations on live clusters.
- Adaptation proposals are generated by `generate_proposals_from_review` and must be reviewed via `check-proposal` before any change is applied.
- Automated CLI commands (e.g. `kubectl apply`) are disallowed unless executed manually by an operator.

## Runtime File Hygiene

- Local runtime configs (`*.local.json`), snapshots, run outputs, and logs are excluded from version control (`.gitignore`).
- Use `runs/`, `snapshots/`, and `logs/` directories for ephemeral data.
- Clean up stale artifacts to avoid retention of sensitive data.

## Review Expectations for Dependency and Provider Changes

- Any addition or update to dependencies, providers, or credential handling code requires a documented security review.
- Such changes must pass through the adaptation proposal workflow, including fixture-based tests and security checks.
